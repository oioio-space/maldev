- The 3 E2E tests in `runtime/bof/example_bofs_windows_test.go`
  (`TestExecute_HelloBeacon_E2E` / `_ParseArgs_E2E` / `_LoadLib_E2E`)
  are gated behind `testutil.RequireIntrusive(t)` and currently
  CRASH the test binary on the win10 TOOLS snapshot with
  `STATUS_ACCESS_VIOLATION` when MALDEV_INTRUSIVE=1. The crash
  is in BOF entry-point execution after the loader stage (loader
  + Beacon API + dollar-import resolver all work — proven by
  the unit tests). Probable causes to investigate on Windows:

  1. The `.o` binaries built with mingw `-ffreestanding
     -fno-stack-protector` may emit prologue code that touches
     the stack outside what `syscall.Syscall` allocates. Try
     building with `-mno-red-zone` (BOFs traditionally use the
     128-byte red zone for scratch).

  2. The COFF symbol resolution may pick a non-`.text` symbol
     for "go" if the C source emits multiple `go` symbols.
     `dumpbin /symbols hello_beacon.o` (or `objdump -t`) on
     Windows would confirm the entry symbol's section number.

  3. The `syscall.Syscall(addr, ...)` call assumes Windows x64
     calling convention but mingw might emit System V code if
     a flag was forgotten. Sanity check with `objdump -d`.

  Recommend: triage on Windows with WinDbg attached
  (`go test -count=1 -run TestExecute_HelloBeacon_E2E
  ./runtime/bof` under WinDbg), or add `GOTRACEBACK=crash`
  to capture the full panic context.

- The `runtime/bof/testdata/loadlib.c` example tests the
  dollar-import path; on the Win10 TOOLS snapshot the actual
  LoadLibraryA call may be hooked by the snapshot's residual
  state — the test accepts both "loaded" and "returned NULL"
  branches deliberately. Test harness comments capture this.
