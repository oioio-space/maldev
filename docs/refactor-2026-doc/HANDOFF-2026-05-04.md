- The 3 E2E tests in `runtime/bof/example_bofs_windows_test.go`
  (`TestExecute_HelloBeacon_E2E` / `_ParseArgs_E2E` / `_LoadLib_E2E`)
  are gated behind `testutil.RequireIntrusive(t)` and currently
  CRASH the test binary on the win10 TOOLS snapshot with
  `STATUS_ACCESS_VIOLATION` when MALDEV_INTRUSIVE=1. **Root cause
  diagnosed Linux-side; fix is well-scoped:**

  The BOF emits `mov rdi, [rip+disp32]` for `__imp_BeaconPrintf`
  references — a `REL32` (32-bit signed RIP-relative) addressing
  mode. Verified via `x86_64-w64-mingw32-objdump -d
  hello_beacon.o`:
  ```
  7:  48 8b 3d 00 00 00 00   mov  0x0(%rip),%rdi
                  a: IMAGE_REL_AMD64_REL32  __imp_BeaconPrintf
  ```

  **Status update (commit `<this-session-tail>`)**: import-table
  co-allocation IS now implemented. `Execute` pre-scans
  relocations via `collectImports`, extends `VirtualAlloc` to
  `len(textData) + 8 * num_imports`, fills slots inline, and
  patches `__imp_<name>` relocations to slot addresses within
  the same allocation. The BOF no longer crashes — the access
  violation at `mov rdi, [rip+disp32]` is gone, BeaconPrintf is
  reached, and bytes flow back into `b.output`.

  **Remaining blocker (next layer):** the BOF's
  `lea rdx, [rip+disp32]` references to `.rdata` (the
  "hello from BOF\n" string literal) are mis-resolved because
  the loader only maps `.text` into memory. `.rdata` is never
  loaded, so cross-section relocations (REL32 from `.text` to
  `.rdata` symbol) target the wrong address — typically a
  point inside the loaded `.text` itself. The HelloBeacon E2E
  fails with output `"WVSH\x83\xec0H\x8b=R"` — those are the
  BOF's own prologue bytes (push rdi; push rsi; push rbx; sub
  $0x30,%rsp; ...) being read as the format string.

  **Fix path (estimated ~80-120 LOC, focused half-session):**
  generalise the loader to map every section that has data
  (`.text`, `.rdata`, `.data`, `.bss`), not just `.text`.
  Rough shape:

  1. In `Execute`, pre-scan all sections; track each one's
     loaded offset within a single contiguous `VirtualAlloc`
     (still keeping the import table at the tail).
  2. Track `sectionBase map[int]uintptr` (1-based section index
     → loaded base address).
  3. In `applyRelocations`, when `sym.SectionNumber > 0`, use
     `sectionBase[int(sym.SectionNumber)] + sym.Value` instead
     of `textBase + sym.Value`.
  4. Watch protection: `.rdata` should be `PAGE_READONLY`
     after copy, but mixing PAGE_EXECUTE_READWRITE for the
     whole page is OK for now (operator-controlled artefact).

  Recommend pickup on Windows: open `runtime/bof/bof_windows.go`
  `Execute`, generalise the section-loop to map all sections
  with raw data, then `MALDEV_INTRUSIVE=1 go test -count=1 -v
  -run TestExecute_HelloBeacon ./runtime/bof` — output should
  read `hello from BOF\n` plus the first arg byte (or no arg
  byte if invoked with `nil` args).
